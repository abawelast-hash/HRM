# الهندسة المعمارية — مشروع ShamIn

## نظرة عامة

مشروع ShamIn مبني على **هندسة طبقات** (Layered Architecture) حيث كل طبقة مسؤولة عن وظيفة محددة ولا تعرف تفاصيل الطبقات الأخرى.

## 2. الطبقات بالتفصيل

### 2.1 طبقة الجمع (Ingestion Layer)

**المسؤولية**: جمع البيانات من مصادر خارجية وتوحيدها

**المكونات**:
- `BaseCollector` — كلاس أساسي يُحدد العقد (Contract): كل جامع يجب أن يُنفذ `collect()`
- `RSSCollector` — يقرأ خلاصات RSS
- `TelegramCollector` — يقرأ رسائل تلغرام مع Anti-Ban
- `WebScraper` — يكشط أسعار من مواقع ويب
- `APICollector` — يجمع مؤشرات اقتصادية من APIs

**نمط التصميم**: **Abstract Factory** — كلاس أساسي ← كلاسات متخصصة

**المُجدوِل** (Celery):
```
Celery Beat → يُرسل مهمة كل X دقائق → Redis (Broker) → Celery Worker → يُنفذ المهمة
```

**التواصل مع الطبقة التالية**: يُرسل بيانات بالتنسيق الموحد إلى طبقة المعالجة

---

### 2.2 طبقة المعالجة (Processing Layer)

**المسؤولية**: تنظيف النصوص واستخراج المعلومات المفيدة

**المكونات**:
- `TextCleaner` — 8 خطوات تنظيف نصي
- `ArabicTokenizer` — تقطيع النص إلى كلمات
- `NumericExtractor` — استخراج أسعار ونسب واتجاهات
- `ProcessingPipeline` — يُنسق عمل المكونات الثلاثة

**نمط التصميم**: **Pipeline Pattern** — سلسلة عمليات متتالية

```
raw_text → TextCleaner.clean() → NumericExtractor.extract_all() → نتيجة معالجة
```

**التواصل**: يُرسل البيانات المعالجة لطبقة التخزين

---

### 2.3 طبقة التخزين (Storage Layer)

**المسؤولية**: حفظ واسترجاع البيانات بكفاءة

**المكونات وتخصصاتها**:

| المكون | التخزين | التخصص |
|--------|---------|--------|
| `relational_db.py` | PostgreSQL | بيانات بنيوية (أخبار، تنبؤات) |
| `timeseries_db.py` | InfluxDB | سلاسل زمنية (أسعار) |
| `object_store.py` | MinIO | ملفات كبيرة (نماذج) |
| `model_registry.py` | JSON محلي | سجل إصدارات النماذج |

**نمط التصميم**: **Repository Pattern** — كل مخزن يُخفي تفاصيل التخزين

**Singleton Pattern**: `get_engine()` و `get_session()` يضمنان إنشاء محرك واحد فقط

**Lazy Initialization**: عملاء InfluxDB و MinIO لا يتصلون إلا عند أول استخدام

---

### 2.4 طبقة الميزات (Features Layer)

**المسؤولية**: تحويل البيانات المعالجة إلى ميزات رقمية يفهمها النموذج

**سيحتوي**:
- `time_features.py` — ميزات زمنية (ساعة، يوم، شهر، عطلة، رمضان)
- `event_features.py` — ميزات أحداث (وزن عسكري، سياسي، اقتصادي مع اضمحلال)
- `embedding_features.py` — تمثيلات نصية رقمية

**التواصل**: يُنتج DataFrame جاهز لإدخاله للنماذج

---

### 2.5 طبقة النمذجة (Models Layer)

**المسؤولية**: التنبؤ بالأسعار المستقبلية

**النماذج**:
1. **TFT** — نموذج رئيسي (تعلم عميق)
2. **XGBoost** — نموذج مساعد (تعلم آلي)
3. **Holt-Winters/ARIMA** — نموذج إحصائي أساسي
4. **Sentiment LSTM** — تحليل مشاعر

**نمط التصميم**: **Strategy Pattern** — كل نموذج يُنفذ نفس الواجهة (`train()`, `predict()`)

---

### 2.6 طبقة الاستدلال (Inference Layer)

**المسؤولية**: تجميع تنبؤات النماذج الثلاثة بأوزان ديناميكية

**المعادلة**:
```
التنبؤ = w₁×TFT + w₂×XGBoost + w₃×HLT
```
حيث w₁+w₂+w₃ = 1.0 والأوزان تتحدث حسب الأداء

---

### 2.7 طبقة العرض (Presentation Layer)

**المسؤولية**: عرض النتائج للمستخدم

**المكونات**:
1. **FastAPI** — واجهة برمجية للتكامل مع أنظمة خارجية
2. **Streamlit** — لوحة بيانات تفاعلية بصرية

---

## 3. البنية التحتية (Infrastructure)

### 3.1 Docker Compose — تنسيق الخدمات

```yaml
الخدمات:
├── shamin_postgres (منفذ 5432) ← قاعدة PostgreSQL
├── shamin_influxdb (منفذ 8086) ← قاعدة InfluxDB  
├── shamin_redis (منفذ 6379) ← وسيط Redis
├── shamin_minio (منفذ 9000/9001) ← تخزين MinIO
├── shamin_celery_worker ← مُنفذ المهام
└── shamin_celery_beat ← مُجدوِل المهام
```

### 3.2 الشبكات والوحدات

- جميع الخدمات على شبكة Docker واحدة (`shamin_network`)
- كل خدمة لها وحدة تخزين مستقلة (Volume) لحفظ البيانات عند إعادة التشغيل
- الوحدات: `shamin_pg_data`, `shamin_influx_data`, `shamin_redis_data`, `shamin_minio_data`

### 3.3 متغيرات البيئة

تُقرأ من `.env` وتُحقن في Docker وفي الكود:

| المتغير | الوصف | مثال |
|---------|-------|------|
| `POSTGRES_URL` | رابط PostgreSQL | `postgresql://shamin:pass@localhost:5432/shamin_db` |
| `INFLUXDB_URL` | رابط InfluxDB | `http://localhost:8086` |
| `INFLUXDB_TOKEN` | رمز InfluxDB | `shamin-dev-token` |
| `INFLUXDB_ORG` | منظمة InfluxDB | `shamin_org` |
| `REDIS_URL` | رابط Redis | `redis://localhost:6379/0` |
| `MINIO_ENDPOINT` | رابط MinIO | `localhost:9000` |
| `MINIO_ACCESS_KEY` | مفتاح MinIO | `shamin_minio` |
| `MINIO_SECRET_KEY` | سر MinIO | `shamin_minio_secret` |
| `TELEGRAM_API_ID` | معرف Telegram | `12345` |
| `TELEGRAM_API_HASH` | بصمة Telegram | `abcdef...` |

## 4. أنماط التصميم المستخدمة

| النمط | أين | الغرض |
|-------|-----|-------|
| **Abstract Base Class** | `BaseCollector` | عقد موحد لجميع الجامعات |
| **Pipeline** | `ProcessingPipeline` | تسلسل عمليات المعالجة |
| **Singleton** | `get_engine()` | منع إنشاء اتصالات متعددة |
| **Lazy Initialization** | `ObjectStore.client` | تأخير الاتصال لحين الحاجة |
| **Strategy** | النماذج الثلاثة | خوارزميات قابلة للتبديل |
| **Repository** | طبقة التخزين | إخفاء تفاصيل التخزين |
| **Observer** | نظام التنبيهات | إشعارات عند أحداث معينة |
| **Factory** | إنشاء الجامعات | إنشاء الجامع المناسب حسب النوع |

---

## 5. تدفق الاعتماديات (Dependency Flow)

```
Presentation → Inference → Models → Features → Processing → Storage ← Ingestion
                                                                ↑
                                                          Scheduler (Celery)
```

**القاعدة**: الاعتماديات تتجه من الأعلى للأسفل فقط — طبقة المعالجة لا تعرف عن طبقة العرض.

## 6. التوسعة المستقبلية

### إضافة مصدر جديد:
1. أنشئ كلاس يرث `BaseCollector`
2. نفِّذ دالة `collect()`
3. أضف إعدادات في `sources.yaml`
4. أضف مهمة في `scheduler.py`

### إضافة نموذج جديد:
1. أنشئ كلاس يُنفذ `train()` و `predict()`
2. أضف إعدادات في `model_config.yaml`
3. أضفه لنظام التجميع مع وزن

### إضافة نقطة نهاية API:
1. أضف route في `src/presentation/api/main.py`
2. عرِّف Pydantic model للمدخلات والمخرجات
